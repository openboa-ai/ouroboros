# MLP-01 Scope And Cutline

## Purpose

This page is the scope-defense document for MLP-01.

Its job is not to list everything that might be useful.

Its job is to defend one believable delegated live path from dilution.

It should let the team reject attractive but off-mission work quickly, consistently, and without
re-opening product identity every time.

## Scope Thesis

MLP-01 is only trying to prove one thing:

**a weak human operator can delegate one serious trading path into believable live operation and
still remain in control.**

That means scope is not defined by:

- what sounds generally valuable
- what creates future platform breadth
- what one subsystem could build next

Scope is defined by:

- what strengthens one believable delegated live path
- what closes the operator's remaining trust questions
- what protects the lovable proof from becoming shallow, broad, or fake

Anything that does not materially strengthen that proof should be treated as out of scope until the
proof lands.

## In-Scope Proof Prerequisites

These are mandatory for MLP-01 because each one unlocks part of the lovable proof.

| Proof prerequisite | Trust question it closes | Why it is required now |
| --- | --- | --- |
| One serious agent-originated path | "Is there anything here worth following?" | The first proof starts with agent leverage, not commentary |
| One durable candidate object | "Is this path real?" | Without this, the operator still carries the record manually |
| Counted versus non-counted evidence visibility | "Why should I trust this path?" | Trust cannot form if legitimacy stays opaque |
| One explicit per-candidate live gate | "What exactly am I approving?" | Promotion meaning must stay serious and bounded |
| One real live path on Binance BTC perpetual futures | "Can this really trade?" | Without real live behavior, the proof stays incomplete |
| Full autonomous execution within explicit limits | "Can I let it run without shadowing it?" | The product must feel alive after promotion |
| Meaningful wake, inspect, pause, stop, and override surfaces | "Can I still stay in control?" | Trust requires credible intervention, not just live execution |

These are not a generic feature checklist.

They are the minimum proof-bearing surfaces for one believable delegated live path.

## Hard Rejects

The following are actively off-mission for MLP-01.

They are not just “nice later.”

They are dangerous now because they weaken proof depth before trust lands.

| Item | Why it sounds attractive | Why it is a hard reject now |
| --- | --- | --- |
| Multi-venue routing | Feels more serious and flexible | Adds breadth before one venue proves depth |
| Assets beyond Binance BTC perpetual futures | Looks like broader market relevance | Weakens the first live proof with extra market surface |
| Team approvals and collaboration workflow | Sounds enterprise-ready | Introduces organizational complexity before solo-operator trust is solved |
| Broad public APIs | Suggests platform posture | Optimizes for future extensibility over current lovable proof |
| Generic plugin ecosystems | Feels like strategic leverage | Expands surface area without strengthening the core path |
| Portfolio-level coordination across many strategies | Sounds more powerful | Moves the product away from one serious delegated path |
| Broad analytics suites | Looks sophisticated | Risks creating dashboards that preserve manual burden rather than replacing it |

## Later Candidates

These are valid later candidates, but they do not belong in MLP-01.

| Later candidate | Why it is legitimate later | Why it is not required now |
| --- | --- | --- |
| Additional crypto venues | Portability matters eventually | First proof should go deep on one venue |
| Additional instruments | Broader coverage may matter later | One believable live path matters more than broader market shape |
| Team workflow and shared approvals | Multi-user trust may matter later | First proof is still one-operator trust |
| Broader external integrations | Useful for ecosystem fit | They do not strengthen the first lovable proof directly |
| Richer ranking, analytics, and optimization layers | Could improve operator leverage later | They are downstream of proving trustworthy delegation |

These should stay out of MLP-01 unless the lovable proof itself is already secure.

## False-Positive Scope

These requests often sound smart but weaken the current proof.

| Proposal | Why it sounds attractive | Why it weakens the proof now |
| --- | --- | --- |
| “Make it venue-agnostic now” | Feels architecturally elegant | Delays first-venue depth and does not close any immediate trust question |
| “Add more strategy coverage” | Creates apparent product breadth | Increases idea surface without proving trustworthy delegation |
| “Add more dashboards” | Looks more operationally serious | Can preserve manual supervision instead of reducing it |
| “Add broader approval features” | Feels safer and more governed | Introduces team-workflow shape before one operator trust loop works |
| “Expose everything as a flexible platform” | Sounds future-proof | Shifts focus from lovable proof to speculative platform positioning |

The test here is simple:

if the idea expands apparent breadth more than it strengthens one believable delegated live path,
it is a false-positive scope item.

## Team Gate Rule

Every proposed addition to MLP-01 must pass this gate.

The team must be able to answer all four questions clearly:

1. Which proof prerequisite does this strengthen?
2. Which trust question does this help close?
3. Which release slice does this help land?
4. Why must this exist before one believable delegated live path is already proven?

If those answers are weak, the item does not enter MLP-01.

The burden of proof is on inclusion, not on rejection.

## Architecture Invariants Scope Cannot Violate

Even the smallest acceptable cut must preserve these invariants:

- stage boundaries stay explicit
- evidence legitimacy boundaries stay explicit
- durable truth stays outside the runtime
- wake authority stays above runtime execution
- adapter-friendly seams survive without delaying first-venue depth
- operator-visible audit and intervention surfaces remain first-class

These are not reasons to include work.

They are constraints that in-scope work must not violate.

## Cutline Tests

### MLP-01 is too small if

- it cannot reach one believable live path
- it cannot explain why a path counted
- it cannot support safe delegation
- it cannot support meaningful intervention

### MLP-01 is too large if

- breadth grows before depth of trust lands
- organizational complexity arrives before solo-operator trust is solved
- market or platform ambition outruns one lovable proof
- the team is building future positioning instead of current proof quality

## Scope Defense Rule

When a new idea is proposed, the team should be able to say:

- what proof it strengthens
- what trust question it helps answer
- which slice it helps land
- why it matters before the lovable proof is already secure

If the idea mostly strengthens future breadth, future platform shape, or future analytics while the
current proof remains weak, it should stay out.
