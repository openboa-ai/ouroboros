## Discovery Highlights

- rigid fixed-frequency or fixed-duration trading rules were rejected
- evaluation should emphasize adequate trade samples instead of hard calendar thresholds
- the app should stay adaptive rather than over-constrained
- important data should be ingested broadly, then filtered by the agent
- model degradation should normally block new entries before stronger intervention
- execution-core invariant failures should trigger execution-core overrides
- self-improvement should run in the background continuously
- dependency expansion should be separated from the default improvement loop
- researcher self-improvement should improve future promoted trader versions instead of directly mutating the current live trader
- artifact thinking should probably bundle researcher and trader as one versioned strategy package, while still keeping the live runtime operationally separated from active research
- the rollback and comparison unit should be the full strategy artifact, not only the trader code
- artifact management should stay portfolio-level across the supported symbols instead of splitting BTC and ETH into separate artifact lineages
- portfolio-level artifact management does not mean synchronized symbols; BTC and ETH decisions should still be able to act independently inside one artifact
- a likely next direction is to treat the strategy artifact as a declarative package, JSON-centered, so agent topology and constraints can stay stable while implementations evolve
- the strategy artifact should likely be a whole directory bundle, and the agent workspace itself should be treated as the primary asset boundary
- semantic / episodic / procedural memory should not be hard-coded as first-class storage categories in the asset contract
- each agent should be free to organize its own workspace materials inside the asset, as long as the workspace remains exportable, inspectable, and replayable
- the bundle should be fully self-contained, but not necessarily collapsed into one giant JSON file; `strategy.json` can stay as the canonical entrypoint while referencing local files inside the same bundle
- `strategy.json` should remain the single canonical entrypoint at the workspace root
- `strategy.json` should behave as a thin bootstrap/index file rather than a giant all-in-one configuration dump
- the top-level shape of `strategy.json` should be organized around `active` and `indexes`
- the minimum `strategy.json.active` set should include `live_lane_ref`, `current_checkpoint_ref`, and `export_policy_ref`
- official clients should not use direct workspace reads/writes as their primary integration path
- official clients should go through a service/application layer that owns validation, invariants, migrations, locking, and export rules
- human inspection of the workspace is still allowed, but machine clients should treat it as internal state behind the app/service boundary
- mutable memory should keep evolving between promotions, but each promotion should freeze an immutable checkpoint snapshot inside the asset
- one artifact should support multiple mutable runtime lanes rather than one shared mutable state
- mutable runtime lanes should stay strongly separated by purpose:
  - one authoritative `live` lane
  - multiple `paper` lanes
  - multiple `shadow` lanes
  - optional read-mostly `observer` lane
- the `live` lane should include the full mutable trading context that live decisions rely on, including live memory, sessions, positions, orders, and evaluation summaries
- live-facing evaluation summaries should retain references to underlying raw evidence rather than existing as summary-only black boxes
- live-lane `positions` and `orders` should preserve both current state and event history
- non-live lanes should be treated as internal runtime/evaluation machinery unless their outputs are actually consumed by the live trading context
- the exportable asset model should stay live-centered even if the local runtime is multi-lane internally, and all artifacts that materially participate in live trading should be export targets
- checkpoint snapshots should be full copies rather than only diff-based metadata so rollback and inspection stay straightforward
- promotion checkpoints should include the paper-trading evidence that justified promotion, while keeping that paper evidence structurally separate from live mutable state
- paper evidence should remain rich, but anything saleable or exportable must exclude user-identifying data, secrets, and other non-shareable material, which suggests a local-master-asset versus sanitized-export distinction
- secrets, credentials, and other host-level trust material should remain outside the asset workspace even if the workspace itself is the primary artifact boundary
- sanitized export should be treated as a first-class capability of the asset model
- sanitized export should be available through a direct app workflow rather than treated as a manual post-processing step
- export targets should include both the current live asset and any specific checkpoint
- export should always run against a checkpoint; exporting the current live asset should first create a fresh checkpoint and then export from it
- checkpoints should have at least three types: promotion, export, and incident
- incident checkpoints should be generated automatically by the system
- promotion checkpoints should be created when a candidate passes paper trading
- live promotion should be decided from checkpoint history and time-series performance, not from a single latest checkpoint only
- promotion analysis should support both recent and cumulative views instead of forcing one weighting scheme
- promotion should be a multi-metric agent judgment, not a single mandatory score threshold
- the evaluation model should have fixed mandatory core dimensions plus agent-adjustable secondary metrics, rather than being either fully frozen or fully mutable
- the fixed core dimensions should explicitly include incident frequency and forced-intervention history
- exhaustive intervention taxonomy is unrealistic; the system should preserve minimal structured incident envelopes plus rich raw logs that the agent can interrogate later
- the evaluator should be self-improving too
- rejected paper candidates should continue in shadow evaluation for a while so evaluator quality can also be judged over time
- shadow evaluation should apply by default to rejected paper candidates
- shadow-evaluation duration should be fixed like paper-evaluation duration for the current artifact policy
- every artifact should remain backtestable on demand
- backtestability means the artifact should be replayable against historical raw market data whenever that data is available
- raw historical data should live in a local shared data store inside the workspace boundary, while artifacts reference collections by stable hashes or snapshot ids
- one shared local raw-data store is enough by default; per-asset private raw-data stores are unnecessary in the current model
- the shared local data store should cover exchange data and other strategy-relevant modalities such as external text and on-chain data
- the shared local data store should have both raw and canonicalized layers
- the canonicalized layer should use app-wide canonical schemas rather than asset-specific private schemas
- a likely direction is to avoid exhaustive source-by-source schemas and instead use:
  - a minimal shared canonical envelope
  - source-specific rich context blobs
  - raw preservation for replay and re-interpretation
- the minimal shared canonical envelope should stay source-centered:
  - source/provenance identity
  - event or publication time
  - local ingest time
  - content identity
- the current minimum source envelope is:
  - `source_ref`
  - `event_time`
  - `ingested_at`
  - `content_hash`
- source entries should stay metadata-centered and point to separate content blobs/files for large article bodies, transcripts, or raw payloads
- symbol linkage and other market interpretation should not be stored on the source entry and do not need their own canonical data layer; they can live in LLM/session/evaluation logs
- the storage model should be workspace-first rather than file-first or DB-first
- a local embedded DB owned by the workspace should count as part of the asset rather than as an external dependency
- the preferred v1 storage shape is hybrid:
  - embedded DB for mutable operational state, indexes, and references
  - blob/files for large immutable bodies, exports, and dataset files
- the current storage preference leans toward an embedded NoSQL/document-oriented store rather than a relational embedded store
- the preferred embedded NoSQL style should follow a document model
- future storage backends may include document stores such as MongoDB, but only as alternative implementations or mirrors of the same storage contract rather than as the defining source of truth
- the storage contract should currently be thought of in five top-level entity types:
  - `artifact`
  - `checkpoint`
  - `collection`
  - `entry`
  - `blob`
- v1 `artifact_id` should use the `UUIDv7` format
- `artifact` should also carry a human-facing `slug`
- the human-facing `slug` should behave more like a renameable alias than a permanent primary key
- `checkpoint` should be first-class and directly addressable, while still remaining attached to a parent `artifact` lineage
- v1 `checkpoint_id` should use the `UUIDv7` format
- human-friendly naming for checkpoints should use optional tags or aliases rather than replacing the primary id
- `collection` should also be first-class and globally reusable rather than owned by one artifact lineage
- `collection` should stay the logical storage unit, even if the physical representation changes later
- in the v1 file-first model, one `collection` should materialize as one `source + UTC hour` shard
- v1 `collection_id` should use the `UUIDv7` format
- each `entry` should have exactly one owner `collection`
- cross-collection reuse should happen by reference rather than by assigning one entry to multiple collections
- `entry_id` should not be content-derived; one blob may be referenced by multiple distinct entries with different source metadata or ingest context
- v1 `entry_id` should be an ingest-assigned immutable id that is stable and opaque rather than a digest of the body
- v1 `entry_id` should use the `UUIDv7` format
- `blob` should be immutable and content-addressed
- multiple entries may reference the same blob when the underlying content is identical
- blob identity should come from content so deduplication and integrity verification are part of the storage model itself
- v1 blob identifiers should use a self-describing string format like `sha256:<lowercase-hex>`
- blob ids should stay future-compatible through the general pattern `algorithm:digest`
- export should bring along the collection references needed for reproducibility, and when appropriate may also include collection material in the exported package
- collection identity should include at least:
  - `kind`
  - `time_range`
  - `content_hash`
- canonical collections should additionally carry:
  - `parent_collection_ref`
  - `transform_version`
- current format direction for file-backed parts of the hybrid model:
  - JSON for top-level manifests
  - NDJSON with the `.ndjson` extension for append-friendly catalogs and event streams
  - JSON for small metadata objects
  - Parquet reserved for large tabular/time-series datasets
- people-facing declarative files should prefer JSON as well, to reduce format sprawl inside the asset model
- avoid mixing both `.jsonl` and `.ndjson`; standardize on `.ndjson`
- use `UTC hour` as the uniform storage partition across sources
- keep `event_time` as the semantic time in the entry; UTC-hour partitioning is only the filesystem/storage rule
- immutable source-body blobs should also be surfaced as workspace documents so collection evidence can be inspected through the same service-owned catalog
- source entries should also be surfaced as first-class workspace documents rather than staying only as lines inside `entries.ndjson`
- blob workspace documents should backlink to the owning collection manifest and entry shard
- blob workspace documents should also backlink to entry documents so source inspection can traverse collection -> entry -> blob
- legacy collection indexes should be normalized into the canonical `items` shape and canonical relative refs during boot so older workspaces keep loading
- protected service-owned roots such as `imports`, `operations`, generated exports, and secret-bearing directories should be physically excluded from checkpoint snapshots and sanitized export workspaces
